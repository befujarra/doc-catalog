📌 Controle de versão

Versionamento ajuda a reverter, alternar entre versões e controlar todas as modificações feitas no projeto.

🧠 Conceitos básicos

  • Repositório — como nos referimos ao projeto como um todo; o diretório raiz a partir do qual serão monitoradas todas as alterações.
  • Commit — é o ato de salvar uma ou mais alterações no código; também pode se referir ao conjunto dessas alterações.
  • Working tree — espaço de trabalho onde você faz modificações. Quando edita, cria ou deleta arquivos, as mudanças acontecem aqui.
  • Staging — área de preparação onde as alterações são separadas para serem salvas em um commit; permite escolher exatamente o que será salvo.
  • Branches — ramificações criadas a partir de um ponto no histórico do projeto para trabalhar paralelamente, sem afetar a ramificação principal.

🔄 Fluxo de trabalho básico

  1. Precisamos inicializar um novo repositório a partir de um diretório.
  2. Em seguida, modificar o conteúdo: criar, modificar e excluir arquivos.
  3. Para salvar, adicionamos as mudanças à área de preparação (stage).
  4. Depois damos commit nas alterações, gravando-as definitivamente no repositório.
  5. Pronto, temos a primeira versão! ✅

Status dos arquivos

U Untracked arquivo novo, não commitado
A Adicionado arquivo na área de stage
M Modificado arquivo rastreado com alterações

⌨️ Comandos básicos

Você precisa estar dentro da pasta do projeto. Se for a primeira vez usando, configure e-mail e nome:

git config --global user.email "seu@email.com"
git config --global user.name "seu nome"

git diffMostra as alterações entre commits, entre commit e working tree, etc.
git mergeIntegra mudanças entre branches.

✅ Boas práticas de commits

  • Nomeie seus commits com cuidado: use o commit -m com uma mensagem clara, explicando o que foi feito, para que quem olhar entenda facilmente.
  • Existe uma convenção de padrões de commits: conventionalcommits.org
  • Revise suas mudanças antes de commitá-las; teste localmente e execute os testes automatizados.
  • Faça commits pequenos e frequentes — a revisão fica melhor, a mensagem mais explicativa e a reversão menos agressiva.
  • Faça um commit após terminar cada pequena alteração.

🏷️ Tipos de versão (Semantic Versioning)

Formato: MAJOR.MINOR.PATCH (ex.: 1.4.2). Cada número tem um papel:

MAJOR

Nova funcionalidade ou mudanças grandes — geralmente quebra compatibilidade.

MINOR

Melhorias e funcionalidades que não alteram o fluxo geral da aplicação.

PATCH

Correções de bugs e pequenas melhorias sem alterar funcionalidades.

🌿 Branches

Branches são ramificações que permitem trabalhar em diferentes linhas de desenvolvimento dentro de um mesmo repositório. Cada branch pode conter commits independentes — os devs trabalham em funcionalidades ou correções sem interferir na branch principal.

🌳 Branch Main / Master = tronco da árvore 🌿 Branches = galhos ligados ao tronco

Elas ajudam a manter o histórico organizado e claro, além de facilitarem a colaboração — cada dev trabalha na sua própria branch e depois integra (merge) as mudanças.

Conceitos de branch

  • Branch principal (Master ou Main) — linha principal com o código limpo, que recebe as mudanças das outras branches.
  • Branch de funcionalidade — criada a partir da principal; você desenvolve a funcionalidade e faz merge após testes positivos.
  • Branch de correção — igual à de funcionalidade, mas voltada para correções de bugs.

Exemplo real

Em projetos é comum ter mais de uma branch principal:

Main Principal (produção)
Qualidade Principal para testes
Desenvolvimento Novas funcionalidades e correções

Trabalhando com branches — fluxo

  1. Entre na branch que deseja clonar: git checkout nome-da-branch
  2. Crie e entre na nova branch: git checkout -b nome-da-branch-nova
    Ou só crie sem entrar: git branch nome-da-branch-nova
    💡 Use git branch para listar todas as branches existentes.
  3. Para integrar as mudanças, entre na branch de destino e execute: git merge nome-da-branch-que-ira-enviar
Outros comandos úteis:
  • git branch --help — mostra todos os comandos de branch
  • git branch -d nome-da-branch — apaga branches que não serão mais usadas
  • git branch -m novo-nome — muda o nome da branch atual

⚡ Conflitos de merge

Ocorrem quando o Git não consegue combinar automaticamente as mudanças de duas branches — geralmente por alterações conflitantes no mesmo trecho de código.

Como identificar conflitos

  • O Git emite erro e informa os arquivos com conflito.
  • Os trechos conflitantes são marcados com: <<<<<<<, ======= e >>>>>>>
  • git merge --abort — cancela o merge em andamento.
  • Após resolver, use git add e git commit para finalizar (ou use as opções da sua IDE).

Como evitar conflitos

  • Comunicação regular com a equipe para evitar trabalhar nos mesmos arquivos e trechos.
  • Pulls frequentes para manter a branch sempre atualizada.
  • Commits pequenos e frequentes.

🌐 Repositórios remotos e GitHub

São versões do projeto hospedadas na internet, que permitem que você e outros devs atuem no mesmo projeto de diferentes locais.

  • Facilitam o trabalho em equipe.
  • Servem como backup do código.
  • Possibilitam integração contínua e outras práticas de DevOps.

O GitHub é uma plataforma de hospedagem de código com interface amigável, pull requests, issues, wikis, kanban e muito mais.

Trabalhando com repositórios remotos

  1. Clonar: git clone url-repo
  2. Enviar commits: git push nome-do-remote nome-da-branch
  3. Sincronizar e atualizar: git pull nome-do-remote nome-da-branch

🔑 SSH e setup inicial de repositório remoto

  1. Crie a pasta do repositório local.
  2. Entre na pasta pelo terminal.
  3. Use git init para iniciar o repositório local.
  4. Crie o README.md para usar como base do primeiro commit.
git add README.md
git commit -m "commit inicial"

Adicione o remoto e faça o push:

git remote add origin https://github.com/usuario/nomeDoRepositorio
git push origin master

Configurando uma chave SSH

No GitHub, em Settings → SSH and GPG keys, você registra chaves para validar o git push. Para criar a chave no terminal:

ssh-keygen -C "email"

Escolha o diretório e crie uma senha. Depois adicione a chave ao agente SSH:

ssh-add caminho-do-arquivo

Isso salva a chave privada no diretório e gera a chave pública para registrar no GitHub. Ao clonar, use o modo SSH.

🍴 Fork e contribuições em projetos

O fork funciona como um clone de um repositório para a sua conta — você copia o repositório mantendo a referência ao projeto raiz. Assim, é possível contribuir para o projeto original, prática muito usada em código aberto.

🚫 Usando o .gitignore

Esse arquivo indica quais arquivos ou tipos de arquivos não devem ser considerados pelo Git (ex.: node_modules, .env).

  • Crie o arquivo na raiz do projeto com o nome .gitignore e indique o que deve ser ignorado.
  • O Git emitirá erro ao tentar adicionar esses arquivos à stage.
  • Para manter uma pasta vazia monitorada, crie dentro dela um arquivo vazio chamado .gitkeep.

🤝 Fluxo colaborativo

  1. Clonar o repositório ou fazer o setup inicial.
  2. Criar uma branch separada para a tarefa, a partir de uma branch inicial.
  3. Mudar para a nova branch (git checkout) e fazer as alterações.
  4. Preparar (git add) e guardar (git commit).
  5. Subir a branch e abrir um Pull Request: git push origin branch
    • GitHub → aba Pull requests → New pull request (ou clique em Compare & pull request).
    • Escolha: base: master ← compare: sua-branch
    • Dê um nome, adicione descrição e um reviewer se necessário.
    • Clique em Create pull request.
    • Sem conflitos e sem aprovação obrigatória: Merge pull request → delete a branch.
  6. O líder aprova (ou não) o PR e integra na branch principal (git merge).
  7. Volte ao passo 2 e repita o loop. 🔁
💡 Dica: Se ficar muito tempo em uma branch, volte à principal e dê git pull; depois retorne à sua branch e faça um merge para puxar as atualizações.
